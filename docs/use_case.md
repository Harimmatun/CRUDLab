# Use Cases: Task Tracking System

## 1. Реєстрація користувача
**Актор:** Гість
**Передумови:** Відсутні
**Основний сценарій:**
1. Гість передає email та пароль.
2. Система перевіряє валідність формату email.
3. Система хешує пароль та зберігає користувача в базу даних.
4. Система повертає підтвердження реєстрації.
**Альтернативні сценарії / помилки:**
- Невалідний формат email -> HTTP 400.
- Користувач з таким email вже існує -> HTTP 409.

## 2. Створення задачі
**Актор:** Авторизований користувач
**Передумови:** Наявність валідного токена автентифікації
**Основний сценарій:**
1. Користувач передає заголовок, опис та ID виконавця.
2. Система перевіряє валідність даних.
3. Система створює нову задачу з базовим статусом "TODO".
4. Система повертає створену сутність.
**Альтернативні сценарії / помилки:**
- Запит без токена або з недійсним токеном -> HTTP 401.
- Пустий заголовок задачі -> HTTP 400.
- Вказаного ID виконавця не існує -> HTTP 404.

## 3. Оновлення статусу задачі
**Актор:** Авторизований користувач
**Передумови:** Задача існує в базі даних
**Основний сценарій:**
1. Користувач передає новий статус задачі ("TODO", "IN_PROGRESS", "DONE").
2. Система оновлює статус відповідного запису.
3. Система повертає оновлену задачу.
**Альтернативні сценарії / помилки:**
- Запит без токена -> HTTP 401.
- Задачі не існує -> HTTP 404.
- Спроба перевести задачу в статус "DONE", якщо вона не має виконавця -> HTTP 400.
