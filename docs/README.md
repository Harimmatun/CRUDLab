# Task Tracking System (CRUDLab)

## Опис
[cite_start]Це базовий REST API сервіс для системи трекінгу задач, розроблений у межах Лабораторної роботи №1 з курсу "Архітектура ПЗ"[cite: 1, 47]. [cite_start]Проєкт реалізує базові CRUD-операції, автентифікацію користувачів та валідацію бізнес-правил (інваріантів)[cite: 17, 19, 24].

## Технічні вимоги
Для коректної роботи проєкту необхідно встановити наступне програмне забезпечення:

* **Python:** версія 3.10 або вище
* **PIP:** менеджер пакетів Python.
* **SQLite:** вбудована база даних.
* **Операційна система:** Windows 10/11, macOS або Linux.

### Основні залежності
* `fastapi`: Фреймворк для побудови API
* `uvicorn`: ASGI-сервер для запуску застосунку
* `sqlalchemy`: ORM для роботи з базою даних
* `pydantic[email]`: Валідація даних та схем
* `python-jose`: Робота з JWT-токенами
* `passlib`: Хешування паролів

## Встановлення та запуск

1.  **Клонуйте репозиторій:**
    ```bash
    git clone <url-вашого-репозиторію>
    cd CRUDLab
    ```

2.  **Створіть віртуальне середовище:**
    ```bash
    # Windows
    python -m venv venv
    venv\Scripts\activate

    # Linux/macOS
    python3 -m venv venv
    source venv/bin/activate
    ```

3.  **Встановіть необхідні бібліотеки:**
    ```bash
    pip install -r requirements.txt
    ```

4.  **Запустіть сервер:**
    ```bash
    uvicorn main:app --reload
    ```
    Після запуску API буде доступне за адресою `http://127.0.0.1:8000`.
    Інтерактивна документація Swagger доступна за адресою `http://127.0.0.1:8000/docs`.

## Питання для самоаналізу

### Архітектура

**1. Де живе бізнес-логіка? Подивіться на ваш код — валідація, перевірка інваріантів, прийняття рішень відбувається в контролерах, сервісах чи окремих класах?**

У цій реалізації (baseline) бізнес-логіка розподілена. Базова валідація форматів даних (наприклад, перевірка коректності email) покладена на Pydantic-схеми. Складніша логіка прийняття рішень та перевірка інваріантів (заборона переведення задачі у статус DONE без виконавця) реалізована безпосередньо в контролерах (ендпоінтах) у файлі `main.py`.

**2. Як пов'язані моделі БД і бізнес-логіка? Чи використовуєте ви ORM-моделі безпосередньо для бізнес-правил, чи є окремі об'єкти?**

Ми використовуємо ORM-моделі SQLAlchemy безпосередньо в бізнес-логіці. Дані з бази даних відразу потрапляють у контролери, де відбувається їх обробка. Окремого шару доменних об'єктів (Entities), незалежних від бази даних, на даному етапі не впроваджено.

**3. Наскільки легко було б замінити базу даних? Уявіть, що вам потрібно перейти з PostgreSQL на MongoDB — скільки файлів доведеться змінити?**

Завдяки використанню SQLAlchemy, заміна однієї реляційної БД на іншу вимагає лише зміни рядка підключення у `database.py`. Проте перехід на NoSQL (наприклад, MongoDB) буде складним, оскільки потребуватиме повного переписування моделей (`models.py`) та методів доступу до даних (`crud.py`).

**4. Наскільки складно було тестувати? Чи залежать ваші unit-тести від бази даних або фреймворку? Чи можна запустити тести бізнес-логіки без підняття інфраструктури?**

Тестування бізнес-логіки наразі неможливе без підняття інфраструктури (FastAPI та SQLite), оскільки логіка жорстко прив'язана до ендпоінтів та запитів до БД. Наші unit-тести фактично є інтеграційними, оскільки вони залежать від стану бази даних та роботи фреймворку.

### Масштабування

**5. Наскільки легко було б масштабувати ваш сервіс? Уявіть, що навантаження зросло в 100 разів — які місця зламаються першими? Чи є у вашому коді стан (in-memory кеші, глобальні змінні), який заважає запустити кілька інстансів? Які архітектурні рішення ви б змінили?**

Сервіс не має стану в пам'яті (in-memory кешів чи глобальних змінних), що дозволяє запустити кілька інстансів паралельно. Однак, при зростанні навантаження в 100 разів першим місцем відмови стане база даних SQLite через блокування файлу під час запису. Для покращення архітектури варто перейти на PostgreSQL, впровадити Service Layer для винесення бізнес-логіки з контролерів та застосувати патерн Repository для повної ізоляції коду від бази даних.
